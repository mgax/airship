API¶

sarge¶

class sarge.Sarge(home_path)¶

The sarge object implements most operations performed by sarge. It acts as container for deployments.

Instances provide a number of Blinker signals:

on_initialize¶

Triggered when a sarge home folder is created. sender is the Sarge instance.

on_activate_version¶

Triggered when a deployment is activated. sender is the Deployment instance. There is one keyword argument: folder - absolute path to new active version folder.

Methods:

get_deployment(name)¶

Retrieve a Deployment by name.

class sarge.Deployment¶

Web application that is deployed using sarge. It has a configuration file and a number of version folders. Only one version is “active” and running.

activate_version(version_folder)¶

Activate a version folder. Creates a runtime folder, generates various configuration files, then notifies supervisor to restart any processes for this deployment.

new_version()¶

Create a new version folder. Copy application code there and then call activate_version().

Plugins¶

The main entry point for a plugin must be a one-argument callable. It gets called at startup and passed in the Sarge instance. At this point it can subscribe to Blinker events.

List of core plug-ins¶

class sarge.NginxPlugin(sarge)¶

Generates a configuration file for each deployment based on its urlmap. Upon activation of a new deployment version, the new nginx configuration is written, and nginx is reloaded.
